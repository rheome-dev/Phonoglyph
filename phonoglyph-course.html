01

What is Phonoglyph?

Music in, video out — powered by the GPU

The core idea

Phonoglyph takes an audio file and turns it into a stunning video where every visual element pulses, glows, and moves in perfect sync with the music. Upload a track, separate it into audio stems, apply visual effects, and export broadcast-quality video.

What makes it special? Instead of animating things the slow way (moving pixels one by one on the CPU), Phonoglyph uses GPU shaders to warp, blend, and light up visuals at 60 frames per second. The result looks like a professional music video — generated automatically from your audio.

1
Upload Audio

Drop in any MP3, WAV, or FLAC file. It's uploaded directly to Cloudflare R2 via presigned URL.

2
AI Stem Separation

The backend splits audio into drums, bass, vocals, and melody using AI-powered source separation.

3
Audio Feature Extraction

Meyda (audio analysis library) extracts RMS, spectral centroid, transients, BPM, and chroma data from each stem.

4
GPU Visualizer

45 shader-based effects respond to audio features. Users map any audio feature to any effect parameter.

5
Export Video

Remotion + AWS Lambda render the exact same visualization deterministically to MP4.

Why the GPU matters

Traditional video editing moves pixels using your CPU. The GPU has thousands of cores that all execute the same instruction simultaneously. For visual effects, this means parallel pixel processing.

CPU Animation

  • Processes pixels sequentially via context.drawImage()
  • GPU→CPU→GPU transfer on every composite
  • Slows to a crawl with 20+ layers
  • Cannot achieve 60fps with complex effects

GPU Compositing (Phonoglyph)

  • All pixels computed in parallel on GPU cores
  • Layers render to GPU textures, composited via shaders
  • Handles 20+ layers at 60fps using render targets
  • Zero CPU readback — pixels never leave the GPU
*
Key Insight

Phonoglyph's compositor never calls context.drawImage(). Each layer renders to its own WebGLRenderTarget (a GPU texture), then a blend-mode shader composites them. The composited result goes through FXAA anti-aliasing and hits the screen canvas. All GPU.

GPU Compositing Pipeline

20+ layers, 60fps — the heart of the visualizer

The layer system

Phonoglyph's visualizer is a stack of layers, each with its own Three.js scene, camera, and render target. They're composited by the MultiLayerCompositor.

BG

Background Layer

Solid color, gradient, or image. zIndex -100. Configured via projectSettingsStore (backgroundColor, isBackgroundVisible).

FX

Effect Layers

45 shader-based effects (bloom, glitch, particles, etc.). Each reads from layers below via accumulation targets. Audio-reactive parameters.

IMG

Media Layers

Images and videos bridged from 2D to GPU via MediaLayerManager. CanvasTexture, VideoTexture, or TextureLoader. Audio-bindable properties.

HUD

Overlay Layers

HUD elements: waveform, spectrogram, stereometer, VU meter, chroma wheel, console feed. Rendered by HudOverlay.tsx (35KB).

The three-step render pipeline

The MultiLayerCompositor executes this exact pipeline on every frame:

1
Render each layer to its GPU texture

For each layer in z-index order: if it needs an accumulation target (e.g., bloom reading layers below), refresh that target first. Then renderer.setRenderTarget(layer.renderTarget), clear with transparent, and renderer.render(layer.scene, layer.camera).

2
Composite layers with GPU blend shaders

Switch to the main render target. For each enabled layer, create a ShaderMaterial with the layer's blend mode (normal, multiply, screen, overlay, add, subtract), set its texture as tDiffuse, and render a full-screen quad. Premultiplied alpha and toneMapped: false are critical for correct intermediate compositing.

3
Post-processing and final output

The composited texture passes through a TexturePass into an EffectComposer with alpha-preserving FXAA. The FXAA shader is modified to output vec4(rgb, texture2D(tDiffuse, vUv).a) instead of vec4(rgb, luma) to preserve transparency. Result: screen canvas.

Blend modes in GLSL

Each layer has a blend mode. The compositor supports six modes, each a separate GLSL shader:

GLSL Blend Shaders 
// Normal — opaque layer on top
gl_FragColor = vec4(texel.rgb, texel.a * opacity);

// Screen — light merges with light
vec3 base = texture2D(tBase, vUv).rgb;
gl_FragColor = vec4(
  1.0 - (1.0 - base) * (1.0 - texel.rgb),
  texel.a * opacity
);

// Multiply — darken
gl_FragColor = vec4(base * texel.rgb, texel.a * opacity);

// Add — pure brightness addition
gl_FragColor = vec4(base + texel.rgb, texel.a * opacity);

// Subtract — remove brightness
gl_FragColor = vec4(max(base - texel.rgb, 0.0), texel.a * opacity);
PLAIN ENGLISH

Normal — If this layer has a pixel, show it. Otherwise, show what's underneath. Like placing a sticker on paper.

Screen — Two projectors on the same screen. Where both are bright, the result is brighter than either alone. Great for glow effects.

Multiply — Dark transparency over an image. Everything gets darker. Great for shadows and vignetting.

Add — Just adds brightness. Used for particles, sparks, and lens flares.

Subtract — Removes brightness. Creates inverse effects, dark masks.

The accumulation target pattern

i
Why some effects need accumulated input

Effects like bloom or ASCII filter don't generate their own content — they transform what's beneath them. The compositor maintains per-layer accumulation targets: GPU textures containing all prior layers blended together. Critical fix: these targets are refreshed before the requesting layer renders, not after, to prevent one-frame-behind artifacts in Lambda renders.

MultiLayerCompositor.ts 
// In the render() method:
for (const layerId of this.layerOrder) {
  const layer = this.layers.get(layerId);
  if (!layer || !layer.enabled) continue;

  // CRITICAL: Refresh BEFORE render, not after
  if (this.accumulationTargets.has(layerId)) {
    this.refreshAccumulationTarget(layerId);
  }

  this.renderer.setRenderTarget(layer.renderTarget);
  this.renderer.clear(true, true, true);
  this.renderer.render(layer.scene, layer.camera);
}
PLAIN ENGLISH

The compositor walks through layers in z-index order. Before rendering each layer, it checks: "Does this layer need to see what's below it?"

If yes, it composites all prior layers into an accumulation texture right now — not from the previous frame. This was a critical bug fix for Lambda renders.

Then the layer renders to its own private render target. The effect's shader can sample from the accumulation texture via this.compositor.getAccumulatedTextureBeforeLayer(this.layerId).

Audio Analysis Pipeline

From raw audio to per-stem features: RMS, spectral centroid, transients, BPM, chroma

The four audio stems

After upload, the backend splits audio into stems. Each stem is independently analyzed for audio features.

D
drums

Transients (sudden volume spikes) — triggers particle bursts, beat-synced slides, punchy impacts. Default decay: 0.3s (fastest).

B
bass

Low-frequency energy — drives slow, heavy movements, color pulses, large-scale warping. Default decay: 0.5s.

V
vocals

Mid-range clarity — triggers medium-speed effects, text reveals, smooth color transitions. Default decay: 0.4s.

M
melody

Harmonic content — high-frequency shimmer, sparkles, fine detail modulation. Default decay: 0.5s.

Audio features extracted per stem

The AudioAnalyzer service (24KB) and the use-audio-analysis hook (17KB) extract these features using Meyda:

rms Root Mean Square — volume/energy level (0.0 to 1.0). The most commonly mapped feature.
spectralCentroid Frequency "center of mass" — brightness. High values = treble-heavy, low = bass-heavy.
spectralRolloff Frequency below which 85% of energy sits. Tracks tonal quality shifts.
loudness Perceptual loudness model. Similar to RMS but weighted for human hearing.
transients Sudden energy spikes. Array of {time, intensity, type?, frequency?}. Drives particle spawns and slideshow advances.
fft Fast Fourier Transform — full frequency spectrum. Used by spectrogram and spectrum analyzer overlays.
chroma 12-bin pitch class distribution (C, C#, D, ..., B). Drives the chroma wheel HUD overlay.
bpm Beats per minute. Detected via onset detection and autocorrelation.

The AudioAnalysisData type

Each stem produces a AudioAnalysisData object stored in the database and passed to the Remotion composition for rendering:

AudioAnalysisData (simplified)

  id: string;
  fileMetadataId: string;
  stemType: 'master' | 'drums' | 'bass'
    | 'vocals' | 'melody';

    frameTimes: Float32Array;

    loudness: Float32Array;
    spectralCentroid: Float32Array;

    chroma: ChromaData[];

  };

  waveformData: {
    points: number[];
    sampleRate: number;

PLAIN ENGLISH

rms, loudness, spectralCentroid — Float32Arrays parallel to frameTimes. One value per analysis frame per feature.

transients — sparse events (not per-frame). Each transient records when it happened and how intense it was. Critical for ParticleNetwork spawning and ImageSlideshow advancing.

waveformData.markers — same transients in a different format, used by the waveform HUD overlay to draw beat markers.

Audio texture: GPU-native feature access

The AudioTextureManager converts the analysis data into a format shaders can read natively.

1
Audio Texture (256 x 64, RGBA Float32)

X axis = normalized time (0 to 1). Y axis = feature row. RGBA channels pack 4 features per row. Feature 0 = Row 0, R channel. Feature 5 = Row 1, G channel. Total capacity: 256 features.

2
Time Texture (1 x 1, RGBA Float32)

Updated once per frame: [currentTime, duration, normalizedTime, 0]. Shaders read texture2D(uTimeTexture, vec2(0.5)).z to get normalized playback position.

3
Feature Metadata Texture (4 x 64)

Stores per-feature metadata: feature index, stem type hash, feature name hash, value range. Shaders can self-discover what each row contains.

GLSL — Reading audio features 
uniform sampler2D uAudioTexture;
uniform sampler2D uTimeTexture;
uniform vec2 uAudioTextureSize; // vec2(256, 64)

float sampleAudioFeature(float featureIndex) {
  vec4 timeData = texture2D(uTimeTexture, vec2(0.5));
  float normalizedTime = timeData.z;

  float rowIndex = floor(featureIndex / 4.0);
  vec2 uv = vec2(
    normalizedTime,
    rowIndex / uAudioTextureSize.y
  );
  vec4 data = texture2D(uAudioTexture, uv);

  float ch = mod(featureIndex, 4.0);
  if (ch < 0.5) return data.r;
  else if (ch < 1.5) return data.g;
  else if (ch < 2.5) return data.b;
  else return data.a;
}
PLAIN ENGLISH

The shader asks: "What time are we at?" The time texture answers: "47.3s out of 210s = 0.225 normalized."

Feature index 5 maps to row 1 (5/4 = 1.25, floor = 1), channel G (5 mod 4 = 1). The UV coordinate is (0.225, 1/64).

It samples the audio texture at that UV. The green channel holds 0.73, meaning 73% of that feature's range.

The shader uses this value to drive something visual: particle speed, glow intensity, color saturation — whatever the user mapped.

The Effects System

45 effects, one abstract base class, 6 categories, ~8,800 lines of GLSL

BaseShaderEffect: the contract

Every effect in Phonoglyph extends BaseShaderEffect. The base class handles all boilerplate; each effect only implements its unique visual logic.

BaseShaderEffect.ts (abstract) 
export abstract class BaseShaderEffect
  implements VisualEffect {

  abstract id: string;
  abstract name: string;
  abstract description: string;
  abstract parameters: Record<string, any>;

  // Standard uniforms (all effects get these):
  // uTexture  — input from compositor
  // uTime     — current time (sec)
  // uResolution — canvas size
  // uMousePos — mouse position

  // Must implement:
  abstract getCustomUniforms(): Record<...>;
  abstract getFragmentShader(): string;
  abstract syncParametersToUniforms(): void;
  abstract updateParameter(name, value): void;

  // Lifecycle (handled by base):
  init(renderer) { /* scene, camera, material, mesh */ }
  update(deltaTime) { /* uTime += dt, sync params */ }
  updateWithTime(absTime) { /* uTime = abs (Remotion) */ }
  resize(w, h) { /* uResolution update */ }
  destroy() { /* dispose geometry/material */ }
}
PLAIN ENGLISH

Every effect declares an id ("bloom"), name ("Bloom"), and a parameters object (intensity, threshold, radius).

getCustomUniforms() returns the uniforms specific to this effect. The base merges them with standard uniforms (uTime, uTexture, uResolution).

getFragmentShader() returns the GLSL code. This is the creative part — the actual pixel manipulation.

Two update modes: update(deltaTime) for live preview (time increments). updateWithTime(absoluteTime) for Remotion (time = frame/fps, deterministic).

If the shader fails to compile, the base class falls back to a magenta material so the error is visible instead of crashing.

Example: BloomEffect

BloomEffect.ts (fragment shader) 
precision highp float;
uniform sampler2D uTexture;
uniform float uIntensity;
uniform float uThreshold;
uniform float uRadius;
varying vec2 vUv;

float luma(vec3 c) {
  return dot(c, vec3(0.299, 0.587, 0.114));
}

void main() {
  vec4 color = texture2D(uTexture, vUv);
  vec3 bloom = vec3(0.0);
  // 5x5 Gaussian-weighted neighbor sampling
  for(float x = -2.0; x <= 2.0; x++) {
    for(float y = -2.0; y <= 2.0; y++) {
      vec2 off = vec2(x, y) * uRadius * 2.0;
      vec4 s = texture2D(uTexture, vUv + off);
      float b = luma(s.rgb);
      float c = smoothstep(uThreshold,
        uThreshold + 0.2, b);
      float w = exp(-length(
        vec2(x,y)) * 0.5);
      bloom += s.rgb * c * w;
    }
  }
  gl_FragColor = vec4(color.rgb + bloom * uIntensity, color.a);
}
PLAIN ENGLISH

For every pixel, sample a 5x5 grid of neighbors. Each neighbor's brightness is compared to the threshold — only bright pixels contribute.

Neighbors are weighted by distance (Gaussian falloff: closer = more influence). The radius parameter controls how far neighbors are sampled.

The accumulated bloom color is multiplied by intensity and added to the original pixel. Bright lights bleed outward.

Resolution-aware: the bloom effect scales sample offsets based on a reference 720px height, so the glow looks consistent at any resolution.

Special effects: stateless rendering

Three effects don't extend BaseShaderEffect because they manage complex internal state. For Lambda rendering, they use pre-computed event arrays instead of per-frame state:

P
ParticleNetworkEffect (838 LOC)

Manages particle lifecycles, connection distances, glow. For Lambda: receives spawnEvents array — pre-computed from transients with time + intensity. Deterministic spawning from seeded random.

I
ImageSlideshowEffect (400+ LOC)

State-machine-based image advancing. For Lambda: receives slideEvents array — pre-computed from transients filtered by sensitivity. Each event triggers the next image. waitForAssets() with 8-second timeout preloads all images before rendering begins.

M
MetaballsEffect (400+ LOC)

MIDI-driven metaball blobs with trails, smoothing, and color palettes. Internal animation state, responds to camera position changes.

The Remotion Render Loop

RayboxComposition.tsx: 1,030 lines that turn store state into video frames

How export works

When you click "Export", the frontend serializes everything — layers, audio analysis, mappings, parameter values, decay times, sensitivities — into composition props. These props are sent to the API, which triggers Lambda.

1
API uploads analysis to R2

Audio analysis data is too large for Lambda payloads. The API uploads it to R2 as JSON and passes analysisUrl instead. Composition props get audioAnalysisData: [] (empty).

2
Lambda selects function + triggers

Picks the Lambda function with highest timeout. Triggers renderMediaOnLambda() with: framesPerLambda: 20, codec: 'h264', chromiumOptions: { gl: 'swangle' }. 3 retries with exponential backoff on rate limits.

3
RayboxComposition initializes

Uses delayRender() to hold rendering. Creates VisualizerManager, registers all effects via EffectRegistry, pre-populates slideEvents and spawnEvents from transients. Fetches analysis from analysisUrl. Calls waitForAssets(). Releases delay when ready.

4
Per-frame render loop (useLayoutEffect)

For each Remotion frame: extract audio features at current time, apply all parameter mappings (feature * modulation * max), inject transient events into particle/slideshow effects, call visualizerManager.update(frame, fps), then gl.flush() + gl.finish() to ensure canvas is ready.

The momentum accumulator: deterministic peaks

When a drum hit fires, we want the visualization to "bump" in a random-feeling direction but remain deterministic. The momentum accumulator model in RayboxComposition achieves this:

calculatePeaksValueStateless() 
function calculatePeaksValueStateless(
  transients, currentTime, decayTime, sensitivity
) {
  let totalMomentum = 0;
  const lookback = decayTime * 3;

  for (const t of transients) {
    if (t.intensity < sensitivity) continue;
    const elapsed = currentTime - t.time;
    if (elapsed < 0 || elapsed > lookback) continue;

    // Deterministic direction from timestamp
    const dir = Remotion.random(`peak-${t.time}`)
      * 2 - 1; // -1 to +1

    // Exponential decay
    const decay = Math.exp(-elapsed / decayTime);
    totalMomentum += dir * t.intensity * decay;
  }

  return Math.max(0, Math.min(1,
    baseValue + totalMomentum));
}
PLAIN ENGLISH

Filter by sensitivity: Only transients above the user's sensitivity threshold contribute. Sensitivity 0.5 = keep the top 50% by intensity.

Lookback window: Only recent transients matter. Lookback = 3x decay time. A 0.3s decay looks back 0.9 seconds.

Deterministic direction: Remotion.random("peak-{time}") always returns the same value for the same seed. Every Lambda instance computes the same direction.

Exponential decay: Each transient's influence fades smoothly. No oscillation (unlike the old damped-spring model). Creates organic drift.

Result: A value between 0 and 1 that "bumps" on beats and drifts smoothly between them. Same on your laptop, same on Lambda #47.

Parameter mapping in the render loop

RayboxComposition — per frame 
// For each mapping (e.g., "layer-1::intensity" -> "drums-rms")
const mapping = mappings[paramKey];
const rawValue = getFeatureValueFromCached(
  cachedAnalysis, fileId, mapping.featureId, time
);

// Modulation knob: 0-1 slider -> -0.5 to +0.5
const knob = Math.max(-0.5,
  Math.min(0.5, mapping.modulationAmount * 2 - 1));

// delta = how much the parameter shifts
const delta = rawValue * knob * maxParamValue;
const final = clamp(baseValue + delta, 0, max);

visualizer.updateEffectParameter(
  layerId, paramName, final
);
PLAIN ENGLISH

Every frame, the loop checks all active mappings. "Is drums-rms mapped to glitch frequency?" Yes — get the current bass RMS value at this time.

The modulation knob (0-1 in the UI) converts to a bipolar range (-0.5 to +0.5). At 0.75, the knob is +0.25 (positive modulation).

Delta = rawValue * knob * maxValue. If bass is at 0.8 and knob is 0.25, delta = 0.8 * 0.25 * max.

Final = base slider position + delta, clamped to valid range. Pushed to the VisualizerManager, which updates the shader uniform.

07

Deterministic Rendering

Why the same frame looks identical on your laptop and in 1,000 Lambda instances

The five pillars of determinism

1
Frame-absolute time

In Remotion, updateWithTime(frame / fps) sets an absolute time. Frame 240 at 30fps = exactly 8.0s. No delta accumulation, no drift.

2
Seeded randomness

Remotion.random("seed-string") returns the same "random" number for the same seed. Particle spawn positions, peak directions, and animation offsets all use deterministic seeds.

3
Software WebGL (swangle)

chromiumOptions.gl = 'swangle' forces software OpenGL. Eliminates GPU driver differences between Lambda instances. Identical floating-point math on every machine.

4
Stateless event arrays

ParticleNetwork and ImageSlideshow use pre-computed spawnEvents / slideEvents arrays instead of per-frame state accumulation. Given the same events + time, output is identical.

5
Accumulation target refresh timing

Effects that read from layers below (bloom, ASCII) get their accumulation targets refreshed before rendering, not from the prior frame. Eliminates stale-texture artifacts in Lambda chunks.

!
Common sources of non-determinism to avoid

Math.random(), Date.now(), performance.now(), GPU driver differences, WebGL context creation order, texture upload timing, and any state that depends on frame ordering (since Lambda chunks render frames out of order).

The overlay types

HUD overlays are specialized visual layers rendered by HudOverlay.tsx (35KB). During export, RemotionOverlayRenderer.tsx extracts the same data from cached analysis instead of live audio.

~

Real-time audio waveform display. Uses RMS values from cached analysis. Beat markers drawn from transient data.

|

Frequency-domain visualization from FFT data. Maintains a 5-frame history buffer at 50ms intervals for smooth scrolling.

O
Stereometer

Lissajous-style stereo field display. Extracts real stereo PCM samples via extractStereoWindow() (1024 samples). Audio decoded via OfflineAudioContext in Lambda.

#
VU Meter

Volume Unit meter showing RMS + peak values. Classic analog-style metering.

C
Chroma Wheel

12-bin pitch class distribution (C through B). Visualizes harmonic content as a color wheel from chroma analysis data.

i
Live vs. Lambda rendering

In the browser, overlays read from the live Web Audio API. In Lambda, RemotionOverlayRenderer fetches the master audio file, decodes it with OfflineAudioContext, and extracts the same data from cached analysis using frameTimes for time-based interpolation.

Development Workflow

Commands, ports, CORS rules, and the Lambda redeploy requirement

Essential commands

pnpm install Install all workspace dependencies. Always use pnpm, never npm.
pnpm dev Start both frontend (port 3000) and backend (port 3001) in dev mode.
pnpm --filter @phonoglyph/web dev Frontend only. Port 3000.
pnpm --filter @phonoglyph/api dev Backend only. Port 3001.
cd apps/web && pnpm remotion studio src/remotion/index.ts --port 3031 Remotion Studio. MUST use port 3031 for R2 CORS compatibility.
!
CORS-allowed ports

Only ports 3000, 3001, and 3031 are whitelisted in the R2 CORS policy. Starting Remotion on any other port will cause R2 fetches to fail silently.

The Lambda redeploy requirement

Changes to files in src/remotion/ or src/lib/visualizer/ affect Lambda renders. After pushing, you must redeploy the Remotion site bundle:

Terminal 
# Redeploy after changing visualizer/remotion files
cd apps/web
npx remotion lambda sites create \
  src/remotion/index.ts \
  --site-name=raybox-renderer

# The serve URL stays the same:
# https://remotionlambda-useast1-...s3.../
# sites/raybox-renderer/
# No code changes needed after redeploy.
WHY

Lambda instances don't pull from git. They download a pre-built bundle from S3. If you change an effect's shader but don't redeploy, Lambda renders will use the old shader code.

The serve URL is hardcoded in the render router. The sites create command overwrites the bundle at the same URL, so no code changes are needed after redeploy.

Key takeaways

1
GPU textures over CPU callbacks

AudioTextureManager packs features into a 256x64 GPU texture. Shaders sample directly. Zero JavaScript-to-GPU communication per frame per effect.

2
The compositor pattern

Separate WebGLRenderTargets per layer + GPU blend shaders = 20+ layers at 60fps. Accumulation targets refreshed before requesting layers for correctness.

3
Determinism is earned at every layer

Frame-absolute time, seeded randomness, software WebGL, stateless event arrays, and fresh accumulation targets. A single Math.random() breaks Lambda renders.

4
BaseShaderEffect scales to 45 effects

One abstract class handles scene, camera, material, uniforms, lifecycle. Each effect only writes its unique GLSL shader and parameter logic. ~8,800 total lines of GLSL.

5
tRPC + Supabase + R2 = full-stack type safety

tRPC gives end-to-end types. Supabase RLS enforces access at the DB level. R2 presigned URLs keep large files off the Express server. 11 routers cover the entire API surface.

6
The render loop is the bridge

RayboxComposition.tsx (1,030 LOC) is where everything converges: store state, audio analysis, parameter mappings, transient injection, and the VisualizerManager. Understanding this file is understanding the whole system.
